# コーディングルール

## ドキュメント・コメント

- クラスおよびメソッドには必ず docstring（XMLコメント `<summary>` 等）を書く
- コメントは日本語で記述し、末尾に「。」は付けない
- 仕様が固まっていない暫定実装には `// TODO:` コメントを残す

## 設計方針

- Unityエンジンで動かすことを前提としたc#で記述する
- ScriptableObject は原則使用しない
  - 変数のみ異なるクラスを定義したい場合は、共通の基底クラスを作り、差分の数だけサブクラスを用意する
- 独自の名前空間（namespace）は使い過ぎない
  - 必要性が明確な場合のみ導入し、細かく分割しない
- Resource.Load()は使用せずAddressablesを用いる
- UIはTextではなくTextMeshProを用いる
  - フォントはAddressablesで"DefaultUIFont.asset"をロードする
- エディタ上での手動操作を要求するような作業の進め方は避ける
- Input System Package（新 Input System）のみ使用し、古いUnityEngine.Inputは避ける

## テスト

- 特別な指示がない限り、実装後はUnity-MCPを用いてコンパイルエラーがないか確認する
- Unity-MCPが無効のときはその旨だけ報告し無効の理由は調査しない

## エラーハンドリング

- 想定外の入力（null など）が来た場合、握りつぶしたり回避したりしない
- 想定外の入力は例外がそのまま発生するようにし、問題を隠蔽しない
